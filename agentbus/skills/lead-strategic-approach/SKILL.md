# lead strategic approach

Use this with the agent-bus workflow when the lead owns the shape of a multi-step or multi-teammate task. It covers the lead's strategic approach; ordinary inbox handling stays in the loop skill and its workflow reference.

## Lead center

The lead keeps a stable center while the work receives pressure from user corrections, teammate reports, prior code, model habits, security concerns, UI taste, and release timing. Keep that center grounded in four things:

- user intent and the defect or drift the user is trying to avoid
- the live artifact, including nearby context and adjacent states
- the expected picture of the completed work
- Key Context as the current shared interpretation for the next run

A lead with a center can change strategy without losing direction. Treat outside suggestions as judgment material, then decide what to reflect, narrow, reject, or turn into follow-up work.

## Expected picture

Before fan-out or patching, form a concrete picture of the result.

- What should the user be able to see, do, read, trust, or decide afterward?
- Where is the change likely to affect nearby alignment, state, wording, data flow, or teammate work?
- What would show that the current plan is drifting?
- What direct evidence would confirm the result without turning verification into ceremony?
- What alternate path becomes better if the observed result differs from the picture?

Update this picture as evidence arrives. Strategy is the lead's ability to revise the path while preserving the work meaning.

## User alignment

Read user wording as a signal about desired work behavior, avoided failure, and tolerance for complexity. Avoid echoing the user's phrase as if it were already the answer. Recover the underlying stake:

- the kind of defect the user is reacting to
- the level of polish or restraint the user expects
- the surrounding artifacts or interactions the user is implicitly comparing
- the point where asking, patching, or deferring would change the user's trust

When the user corrects the lead, classify the correction before acting: actual defect, misunderstood intent, weak report, wrong completion criterion, or a broader operating rule.

## Teammate fan-out

Fan-out separates judgment character. Use it to keep different kinds of judgment from flattening into one generic review.

For each teammate request, name the judgment character and the direct target:

- judgment character: design, architecture, prose, security, workflow, release, research, or a more specific lens
- owned target: file, UI state, command path, claim, behavior, or decision boundary
- adjacent effects: what nearby state may move if the teammate changes or recommends something
- report material: what the lead needs to compare, synthesize, or reject the finding
- continuation condition: when the teammate should leave a follow-up request, wait, mark input_required, or close the slice

A precise request gives the lead material that can be synthesized. Ask for the slice the lead can actually compare and decide on.

## Key Context use

Key Context carries the current strategic interpretation. Keep it focused on work meaning and judgment background.

Update Key Context when:

- user intent is clarified or corrected
- teammate reports change the center of the work
- the expected picture changes after direct inspection
- the next run needs a different judgment lens
- closure should preserve a reusable lesson for the next loop

Keep task lists, status summaries, and general rules in their own records. Key Context should change what the next teammate considers important.

## Causal review

Treat every change as an intervention with nearby consequences.

- A UI value can shift alignment, density, hover, focus, dark mode, and neighboring controls.
- A sentence can shift section role, reader expectation, list independence, and terminology.
- A CLI option can shift help shape, docs, smoke tests, and teammate prompt contracts.
- A security rule can shift dashboard view, packet policy, bridge behavior, logs, and user trust.

After a change, inspect the neighboring state that may have moved. The edited line is only the starting point.

## Synthesis

Synthesis is a lead decision. Teammate reports become useful when the lead compares and decides what they change.

Compare reports against:

- the live artifact and nearby context
- Key Context
- user intent and completion criterion
- the expected picture
- conflicting teammate judgments

Then decide what to reflect, reframe, discard, narrow, or defer. If the result differs from the expected picture, update the strategy before creating the next request.

## Model habit correction

Use the bus to reduce default model habits that make work feel generic. Watch for these signs:

- a keyword from the user turns into broad framing instead of target-specific work
- a balanced bullet structure replaces direct judgment
- a small request becomes a one-value patch without neighboring checks
- teammate wording is copied instead of re-judged
- a report sounds polished while saying little about the live artifact

Correct those habits by returning to the target, naming the concrete cause, and letting the report shape fit the evidence. Human-like variety comes from contact with the actual artifact: a UI misalignment, a section role, a security boundary, a repeated code habit, or a user correction that changes the work meaning.

## Closure and reusable learning

At closure, leave the next loop better prepared.

Record only lessons that will change future work:

- a judgment the lead handled too narrowly
- a causal link that was missed
- a repeated implementation, UI, prose, or coordination habit
- a criterion that should appear earlier in future requests
- a local skill or workflow note worth keeping, simplifying, or retiring

Keep durable project outputs in project files, coordination in bus records, and reusable operating lessons in local skills or workflow guidance.
